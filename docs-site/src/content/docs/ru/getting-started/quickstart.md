Это руководство проведёт вас от чистой установки до запуска Codex с моделью не от OpenAI.

## 1. Запустите мастер настройки

```bash
ocx init
```

`ocx init` проведёт вас по следующим шагам:

1. **Выбор провайдера** — выберите один из 50 встроенных пресетов реестра или `custom`, чтобы
   ввести базовый URL и адаптер вручную.
2. **API-ключ** — вставьте ключ или сошлитесь на переменную окружения вида `${ANTHROPIC_API_KEY}`.
3. **Модель по умолчанию** — для провайдеров с ключом, локальных и `custom` примите значение из
   пресета или введите id модели.
4. **Порт прокси** — по умолчанию `10100`.
5. **Внедрить в Codex?** — при обычной loopback-настройке opencodex добавляет корневой
   `openai_base_url` в `$CODEX_HOME/config.toml` (по умолчанию `~/.codex/config.toml`), чтобы
   встроенный провайдер `openai` в Codex указывал на прокси. При привязке к удалённым/LAN-адресам
   вместо этого используется отдельная запись провайдера с заголовком API-аутентификации.
6. **Установить shim автозапуска?** — если включено, при запуске `codex` сначала выполняется
   `ocx ensure`.

Результат сохраняется в `$OPENCODEX_HOME/config.json` (по умолчанию `~/.opencodex/config.json`).

:::note[Записи каталога GPT-5.6]
Стабильная версия v2.7.1 добавляет GPT-5.6 Sol/Terra/Luna для passthrough ChatGPT, OpenAI по
API-ключу, OpenRouter и экспериментального адаптера Cursor. Они работают только при наличии
доступа у соответствующего вышестоящего аккаунта. Пресеты OpenAI по API-ключу и OpenRouter
заявляют доступное контекстное окно в 372 000 токенов; Cursor использует собственные метаданные
адаптера.
:::

## 2. Запустите прокси

```bash
ocx start            # порт по умолчанию 10100
ocx start --port 8080
```

При запуске opencodex:

- записывает свой PID в `~/.opencodex/ocx.pid` (и отказывается запускаться повторно),
- запрашивает актуальные модели у провайдеров, которые это поддерживают, и **синхронизирует
  нативные и маршрутизируемые записи в каталог моделей Codex**,
- слушает `http://localhost:<port>/v1`.

Если запрошенный порт занят, `ocx start` выбирает свободный порт, записывает его в
`runtime-port.json` и обновляет настройки Codex, чтобы тот использовал актуальный адрес.

Проверьте:

```bash
ocx status
ocx gui       # открыть панель управления на актуальном порту
```

## 3. Используйте Codex

Теперь Codex прозрачно общается с opencodex:

```bash
codex "Refactor this function for readability"
```

Чтобы обратиться к конкретной маршрутизируемой модели, используйте форму `provider/model`,
которую показывает селектор модели в Codex:

```bash
codex -m "anthropic/claude-opus-5" "Explain this stack trace"
codex -m "ollama-cloud/glm-5.2"      "Write a SQL migration"
```

## Выбор моделей подагентов (опционально)

В свежей конфигурации в селекторе подагентов Codex представлены пять нативных моделей: `gpt-5.5`,
`gpt-5.6-sol`, `gpt-5.6-terra`, `gpt-5.6-luna` и `gpt-5.4-mini`. Откройте `ocx gui`, чтобы
заменить или переупорядочить до пяти нативных или маршрутизируемых моделей. В панели управления
также можно задать предпочитаемую модель подагента и уровень рассуждений; opencodex добавляет это
указание в v1-запросы совместной работы.

## Вход в аккаунт вместо вставки ключа

Некоторые провайдеры поддерживают полноценный вход в аккаунт (OAuth с автообновлением):

```bash
ocx login xai          # или: anthropic, kimi, kiro, google-antigravity, cursor
ocx logout xai
```

Самому OpenAI ключ **не нужен** — провайдер по умолчанию напрямую пересылает ваши существующие
учётные данные `codex login` (см. [Провайдеры](/ru/guides/providers/)).

## Остановка и восстановление

```bash
ocx stop          # остановить прокси и восстановить нативный Codex
ocx restore       # восстановить нативный Codex без остановки (псевдоним: ocx eject)
ocx restore back  # снова направить Codex через всё ещё работающий прокси
```
