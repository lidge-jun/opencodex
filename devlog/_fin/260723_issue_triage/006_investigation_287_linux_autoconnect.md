## Linux equivalents and implementation shape

### Shell startup hook: required for reliable plain-terminal behavior

The closest equivalent is to reuse claude-env.sh and install marked source snippets
into the actual interactive shell startup files. Bash has no single universal file:
login Bash reads the first available of ~/.bash_profile, ~/.bash_login, or ~/.profile;
interactive non-login Bash reads ~/.bashrc. A correct implementation must therefore
cover at least Bash login and non-login plus zsh, preserve contents and modes, remove
only its own marker blocks, honor OPENCODEX_HOME, and define unsupported shells such
as fish. Source: [GNU Bash Reference Manual — Bash Startup Files](https://www.gnu.org/software/bash/manual/html_node/Bash-Startup-Files.html).

The env-file renderer is reusable, but the current hook is one hard-coded .zshrc
implementation.

> src/server/system-env.ts:8-15
>

> // Shell-hook env file: written on inject, sourced by the shell hook in .zshrc.
> // This works for ALL new shells immediately, unlike launchctl setenv which only
> // reaches processes launched directly by launchd (not Terminal.app children).

> export function getShellEnvFilePath(): string {

### systemd user environment: optional supplement, not a shell replacement

~/.config/environment.d/*.conf or systemctl --user set-environment can feed
processes spawned by the systemd user manager, but cannot guarantee SSH shells or
terminals launched outside that manager. The upstream manual explicitly limits the
assignments to services started by the user instance and distinguishes other shell
parents. Source: [systemd environment.d manual source](https://github.com/systemd/systemd/blob/main/man/environment.d.xml).

OpenCodex already has a clean Linux/systemd seam for its proxy service, but that
unit only defines the proxy process environment; it does not establish a universal
interactive-shell environment.

### /etc/environment: reject for this feature

/etc/environment is a system/PAM login input, not a per-user, non-privileged dynamic
toggle. It would require elevated writes, broaden the effect beyond one OpenCodex
user, and make ownership-safe revert materially harder. Source:
[Linux-PAM pam_env manual source](https://github.com/linux-pam/linux-pam/blob/master/modules/pam_env/pam_env.8.xml).

The current contract is user-wins and tracks only keys OpenCodex injected, which
should not be weakened by editing a global login file.

> src/server/system-env.ts:257-263

### Is there a clean platform seam?

No. There are reusable pieces (env-file rendering, model-env computation, tracking,
gateway-cache and roster sync), but injection, rollback, ownership checks, and revert
directly call launchctl helpers. Removing the Darwin guard alone would run launchctl
on Linux. Resolution (a) therefore needs a platform-neutral payload/sync layer and
separate Darwin and Linux backends, not a small branch inside one function.

## Resolution comparison

### (a) Implement Linux environment injection

Scope: refactor system-env.ts into platform-neutral payload/sync plus Darwin and
Linux backends; add safe Bash login/non-login and zsh hook install/uninstall;
preserve OPENCODEX_HOME, ownership, user-wins semantics, stale cleanup, rollback,
cache and roster sync; optionally supplement with systemd user environment. Update
docs and add Linux-specific tests.

Risk: medium-high. This edits user dotfiles and may carry an API key in a mode-0600
generated file; shell/login behavior varies; SSH/container/WSL may lack a user bus;
disable/uninstall must not remove user content or leave stale routing; Windows must
remain an explicit no-op.

Test surface: extend tests/system-env.test.ts with Linux hook matrices and injected
filesystem/command dependencies; add Bash subprocess smoke coverage; retain API
tests; exercise Ubuntu CI and, if systemd is involved, the real user-systemd job.
The full suite already runs on Ubuntu, Windows, and macOS.

Estimated effort: 2–4 engineering days plus dotfile/secret-lifecycle review.

### (b) GUI honesty: disable with explanation on unsupported hosts

Scope: return a server-derived capability from /api/claude-code, add a disabled
state to SettingToggle, show a localized “macOS-only; use ocx claude” explanation,
and prevent unsupported saves from looking successful. Disable-with-explanation is
better than hiding because it explains an already-stored true value and provides the
workaround. Browser navigator.platform is the wrong source because a remotely opened
dashboard can run on a different machine from the proxy.

Test surface: management API capability contract, focused GUI state/render coverage
if a harness is added, all locale keys, GUI lint/build, root typecheck, and existing
management/system-env tests. No shell or privilege behavior changes.

Estimated effort: 4–8 engineering hours.

## Verdict

**OpenCodex bug (capability/GUI contract), not upstream Claude Code and not a
needs-repro issue.** The server intentionally supports Auto-connect only on Darwin,
but Linux receives an actionable toggle, persisted true state, a discarded “not
macOS” result, and a success response. Keep issue #287 in Bucket 2; do not reclassify
